# Week 3 Reflection — Creational Patterns

## Pattern chosen: Factory Method

### What it is

The Factory Method pattern defines an interface for creating an object but lets
subclasses (or static methods on the same class) decide which concrete object
to instantiate. The caller asks a factory for an object rather than calling a
constructor directly. This centralizes creation logic and makes it easy to swap
out variants without changing every call site.

### Where it applies in my sprint code

My current project is UVSim, a BasicML simulator. The core class lives in
`uvsim.py`. Its constructor already accepts two optional parameters:

```python
def __init__(self, input_function=None, output_function=None):
    self.input_function = input_function or self._default_input
    self.output_function = output_function or self._default_output
```

This means `UVSim` can run in two distinct modes depending on how it is wired:

- **CLI mode** — uses `_default_input` (reads from `input()`) and
  `_default_output` (prints to stdout). This is what `main.py` does on line 13:
  `sim = UVSim()`.
- **GUI mode** — needs to read from a Tkinter `Entry` widget and write to a
  `ScrolledText` widget. `gui.py` exists but the two classes are not yet wired
  together; the GUI's `handle_enter` method does not yet call into `UVSim` at
  all.

Right now, both entry points create `UVSim` objects ad hoc. When the GUI
integration lands, the construction logic — choosing the right `input_function`
and `output_function` — will have to be duplicated or scattered across files.

### The concrete decision

I would add a `UVSimFactory` with two factory methods before writing any more
GUI integration code:

```python
class UVSimFactory:
    @staticmethod
    def create_cli() -> UVSim:
        return UVSim()

    @staticmethod
    def create_gui(gui_instance) -> UVSim:
        return UVSim(
            input_function=gui_instance.get_input,
            output_function=gui_instance.display_output,
        )
```

`main.py` would change from `sim = UVSim()` to
`sim = UVSimFactory.create_cli()`. The GUI startup would call
`UVSimFactory.create_gui(app)`. Neither call site needs to know which
`input_function` / `output_function` to pass — that knowledge lives in one
place.

### Why Factory Method and not another creational pattern

- **Singleton** does not fit: the GUI will eventually support opening multiple
  simulator windows, so forcing a single shared instance would be wrong.
- **Builder** is overkill: `UVSim` has only two meaningful construction
  variants and no optional component assembly.
- **Prototype** does not apply: there is no need to clone existing simulator
  instances.

Factory Method is the minimum pattern that solves the actual problem — two
creation paths, one canonical place to define each.
