# Week 5 Reflection — Behavioral Patterns

**Pattern: Observer**

The Observer pattern defines a one-to-many dependency between objects so that
when one object (the subject) changes state, all registered dependents
(observers) are notified and updated automatically.

In Sprint 2, UVSim's GUI integration exposed exactly this need. The simulator
core (`uvsim.py`) executes instructions and changes internal state — the
accumulator value, the program counter, the contents of memory. The GUI needs
to reflect those changes in real time: updating a memory display panel,
showing the current accumulator, and highlighting the active instruction line.

The naive approach was to have `uvsim.py` call GUI methods directly, but that
created a hard dependency from the core simulator onto the GUI layer — the
opposite of what we wanted. Observer inverts this: the simulator becomes the
subject, and the GUI registers itself as an observer. Whenever the simulator
finishes executing an instruction, it notifies all registered observers with
the updated state. The GUI observer receives that notification and redraws
only the parts of the interface that changed.

This mattered concretely because UVSim supports both CLI and GUI modes. In CLI
mode there are no observers registered — the simulator runs silently and
nothing breaks. In GUI mode the display registers as an observer at startup.
The core never needs to know which mode it is in; it just calls `notify()` and
lets the observer list handle the rest. The same decoupling that the w03
Factory Method reflection identified at construction time, Observer handles at
runtime.

The pattern also made this week's CC artifact easier to reason about: the
PR Readiness Checker subagent is itself observer-like — it watches a branch's
state and reports changes without being coupled to whatever parent dispatched
it.
