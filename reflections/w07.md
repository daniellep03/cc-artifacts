# Week 7 Reflection — EIPs Part 2 · State Charts

**Concept: Statechart**

A statechart is a state machine extended with hierarchical states, parallel
regions, and history. Rather than an implicit tangle of boolean flags, a
statechart makes every possible situation in a system an explicit, named state.
An event triggers a transition between states; an optional guard checks whether
that transition is allowed; and an action fires as a side effect when the
transition occurs.

UVSim's execution lifecycle is a statechart in everything but name. The
simulator has four states I can now identify precisely: **Idle** (no program
loaded), **Ready** (program loaded, awaiting start), **Running** (executing
instructions), and **Halted** (program terminated or faulted). Events like
`load_program`, `run`, `step`, and an internal `halt_instruction` drive
transitions between them.

Guards appear naturally: the `run` event should only fire the
`Ready → Running` transition when a valid program is actually loaded — a guard
condition, not an ad-hoc if-check buried in the handler. The `step` event is
only valid from `Ready` or `Running`, never from `Halted`.

In Sprint 2, adding the GUI made this implicit state space painful. The
interface needed to enable and disable buttons based on simulator state — Run
only active in Ready, Stop only active in Running. Without thinking in
statechart terms, we ended up checking `is_running` and `is_loaded` flags in a
dozen places and getting inconsistent button states. Encoding the four named
states explicitly and having the GUI react to transitions — with an action like
`enable_run_button` firing on the `Idle → Ready` transition — would have
eliminated that class of bug entirely.

The biggest insight from this week: state should be something you name and
design, not something that emerges from which flags happen to be set.
